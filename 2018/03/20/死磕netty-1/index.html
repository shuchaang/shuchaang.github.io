死磕netty(1)-简介

netty
  • netty

JAVA IO 通信

BIO通信

BIO通信通常会有一个专门的线程Acceptor负责接收客户端的请求，并对每一个请求创建一个线程为其服务。服务完成后通过应答流返回给客户端，最后线程销毁。这就是典型的一应答一请求。由于线程是服务器非常宝贵的一种资源，所以这种模型在大并发的情况先引起线程数量膨胀最终会导致整个服务不可用或者Stack Overflow的错误，所以BIO模型缺乏弹性伸缩的能力。

伪异步通信

伪异步通信当接收到用户请求时，不需要创建新线程，而是把它封装成一个Task,交给线程池来做服务。这样我们就可以通过配置线程池来控制线程创建的数量，线程池问题具体可以看看JDK的Executor框架。但是这种模型不足之处在于，当请求数量庞大，线程池可能会出现阻塞。

NIO通信

先要了解几个关键概念:

  • 缓冲区 Buffer 在NIO模型中，所有数据都是用缓冲区来处理。也就是读取数据从缓冲区读，写入数据是写入缓冲区。
  • 通道 Channel 网络数据都是通过Channel来读取和写入。Channel和Stream的区别在于，Stream只能单向，而Channel是双向的。
  • 多路复用器 Selector 它是java Nio的基础，它提供了选择就绪任务的能力。也就是说selector会不断轮询绑定在其上的channel，如果某个channel上发生了读或写事件，它就会被轮训出来。然后通过selectionKey获取就绪任务的集合进行后续的io操作，由于JDK使用了epoll代替了传统的select实现，所以并没有最大链接数的限制。

AIO通信

AIO是一种异步的通信方式，上面的几种都属于同步模式。当有用户请求进入，AIO会注册它的读写时间和返回函数，当IO操作完成后主动通知用户请求讲数据取走。对应于Unix网络编程中的事件驱动IO

对比

  • 线程数量: BIO(m:m)>伪异步(m:n)>NIO(m:1)>AIO(m:0)
  • IO类型 : BIO(同步阻塞) 伪异步(同步阻塞) NIO（同步非阻塞) AIO(异步)

为什么用netty

  1. 原生NIO有缺陷，类库繁杂不易操作
  2. JDK NIO bug (epoll bug)
  3. netty api简单 并且修复了jdk bug
  4. netty 性能高
  5. netty成熟的使用，如dubbo,springcloud,reddison等大量框架底层都使用netty
