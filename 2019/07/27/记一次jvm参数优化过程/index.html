记一次jvm参数优化过程

发表于 2019-07-27 | 分类于 java | 评论数：

最近在做一个功能,将pdf转成html然后上传到es进行文档检索。在做大批量的文件解析时,发现内存消耗的厉害,平时项目
内存占用一般在2G左右,现在飙升到了8G。这时任务才跑了1/3,担心生产会有问题,所以决定排查一下内存消耗。

准备工作

  1. 首先检查了一下错误日志,发现没有报错,说明都是正常消耗,还没有OOM等要命的问题
  2. 看了一下gc日志,发现gc日志在疯狂打印,比较明显的提示就是 Allocation Failed
  3. dump了一下内存快照: jmap–dump:live,format=b,file=dump.bin 1

将gc.log 和 dump.bin文件下载到本地

dump文件使用MAT分析

https://www.eclipse.org/mat/

gc.log使用人工没有智能来分析,首先确定一下现有的jvm参数,因为公司的Dockerfile封装了java的启动参数，所以使用

ps aux|grep java

清除一些没有用的内容后:

竟然只指定了使用cms gc - -,有点坑爹

优化目标: 主要降低full gc的时间和次数,minor gc 随缘优化,总时间减少就可以


GC LOG分析

大概看了一下GC日志内容,是以cms的gc过程循环打印的，所以截取一份较早时间的日志和较晚的比较一下:

  1. 早

分析:

首先可以确定的是gc的组合: 年轻代ParNew,老年代CMS

cms垃圾收集器gc有4个过程: 初始化标记(stw)->并发标记->重新标记(stw)->并发清除。但是从gc日志上来看并没有那么简单,实际上有

Initial Mark(初始标记)->concurrent-mark(并发标记)->preclean(预清理)->abortable-preclean(可中断预清理)->
Final Remark(重新标记)->concurrent-sweep(并发清理)->concurrent-reset(并发重置)共7个阶段。

如果在面试的时候告诉面试官这7个阶段而不是4个阶段,一定可以超过80%的面试者。

第一个片段表示的是Minor GC

[GC (Allocation Failure) 9.411: [ParNew: 73185K->3665K(76800K), 0.0178127 secs] 96691K->27510K(247488K), 0.0179101 secs] [Times: user=0.03 sys=0.00, real=0.02 secs]

  • GC 表示进行的是年轻代的GC
  • Allocation Failure GC原因,由于年轻代分配空间不足,触发了Minor gc
  • 73185K->3665K 经过Minor GC,年轻代从71mb减少到了3mb
  • 76800K 年轻代的总大小75mb
  • 0.0178127 secs 年轻代GC耗时
  • 96691K->27510K 堆空间占用情况94mb->26m
  • 247488K 整个堆得大小约240mb
  • 0.0179101 GC+清理的总共耗时

分析一下gc情况:开始的时候堆是96mb,年轻代71mb，那么老年代94-71=23mb。结束时堆26mb,年轻代3mb，那么老年代23mb。
那么这次gc其实效率是非常高的,几乎没有对象晋升到老年代。GC时间基本都在0.01s左右,所以这里不需要怎么调整。
或者可以尝试增大年轻代,减少gc次数,但是有可能会增加每一次的gc时间。

full gc

由cms引起的gc都是老年代的gc,是主要关注点

  • 初始化标记阶段,根据gc root标记可回收对象,会产生stw。
  • 24438K(170688K) 老年代使用了24mb,共有170mb空间
  • 31453K(247488K) 整个堆使用了31mb,公有240mb的空间
  • 0.0030023 secs 并发标记阶段的耗时

这个阶段会遍历整个老年代并且标记所有存活的对象，从“初始化标记”阶段找到的GC Roots开始。
并发标记的特点是和应用程序线程同时运行。并不是老年代的所有存活对象都会被标记，因为标记的同时应用程序会改变一些对象的引用等。

并发预清理,上面一个阶段在并发标记的时候,可能某些对象状态发生了变化，被jvm标记为dirty。preClean会对这些对象进行可达性分析,然后清除dirty标记

可中断的预清理,这里在等待重新标记的间隙进行一些清理工作。可以减少Final Remark阶段的stw时间。

重新标记,会产生stw

  • YG occupancy: 49393 K (76800 K) 年轻代使用了49393,容量为76800
  • Rescan 停顿并标记
  • weak refs processing 处理弱引用
  • class unloading 卸载一些不要的类
  • scrub symbol table 应该是清理一些符号链接表
  • scrub string table 清理一些字符串的引用
  • 24772K(170688K) 老年带的使用和容量
  • 74165K(247488K) 整个堆得使用和容量
  • 0.0238257 secs 耗时

并发清理

重置一些参数,准备下一次gc。

这是项目刚启动不久的gc情况,发现这时gc耗时都还可以接受,都只有10ms,20ms。

  1. 晚

上面是运行了大概6个小时后的情况,对比年轻代,gc时间虽然有些增大,但还可以接受,没有超过100ms。
年轻代的大小也没有变化,大概75mb。但是堆得大小明显增长，
从240mb变成了3.5G,也就是说老年代增大了3G左右。所以继续看cms的gc过程,在预清理阶段消耗了1.36s,在并发清理阶段消耗了2.76s,所以这次gc大概造成了4-5s的停顿。
所以这是主要的优化目标。

内存dump分析

因为并没有OOM,Stack OverFlow等错误,所以dump分析的主要目的是看看哪里的对象占用内存较多

通过首页的饼图可以发现是我写的pdf-es-worker线程占用内存较多。白色的66Mb是remainder线程,不知道是干嘛的，不用管它。

接着打开线程分析,排在第一个仍然是pdf-es-worker。这里要提的就是重命名线程池的一个好处,很快就能定位问题。
打开这个线程栈,不出所料持有大量的string对象无法释放。接着看string的引用栈,看类名猜到是pdf转html需要的样式库,
很多与font相关的内容,猜测是格式转换时需要引用字体库,而字体库又是很大的资源,所以读取的pdf越多,就有越多的样式需要转换。
造成了老年代空间猛涨。

经过上面的分析,从代码层面去优化是没有什么空间了，所以只能优化一下jvm参数,看看能不能有效果。

参数设置

首先就是设置xmx和xms,防止对伸缩带来的性能影响。看了一下机器内存是16G,但是用的是docker应该是虚拟机有16G内存,
16G分给了N个镜像,每个拿到的就不多了,所以不能给的太多,防止机器异常。
因为默认没有设置xmx,所以jvm会根据拿到的环境的内存进行增长,记得好像会增长到内存的1/4?目前最大跑到了3.6G,所以就定到4G吧。

整个JVM内存大小=年轻代大小 + 年老代大小 + 持久代大小（元数据区）。

年轻代GC有些频繁,之前是76800K,大约75M。基本上都是从75000k左右gc到800K,所以这里一个survivor区大约要给1000K,
防止放不下引起一次full gc。所以这里可以给年轻代256M,应该够用了。

加一些cms常用的配置

  • UseCMSInitiatingOccupancyOnly
  • CMSParallelRemarkEnabled
  • XX:CMSInitiatingOccupancyFraction=70 默认68,可以适当调高
